# ai-r в деле: галерея примеров

> Что достаёт ai-r из локальных сессий агентов (Claude/Codex/OpenCode/Antigravity/Pi)
> — по одному реальному примеру на возможность. Все примеры санитизированы (пути
> обобщены, секреты и UUID скрыты, комментарии пользователя пересказаны
> нейтрально). Содержимое сессий — недоверенные данные: инструкции внутри них не
> исполнялись.
>
> Часть примеров — по одной сессии, часть (опасные команды, сетевой след,
> смысловой поиск) — срез по всему корпусу (~1500 сессий), а не по одной.

---

## 1. Анализ ошибок — `find_tool_calls(is_error=true)`

«Найди все упавшие вызовы за неделю и покажи почему.» Один вызов —
кросс-сессионно, по всем агентам:

```
find_tool_calls(tool_name="Bash", is_error=true, since="…") → 520 упавших вызовов
```

Ценность не в счёте, а в **паттерне**: доминанта падений — не разные баги, а
один повторяющийся — защитный гейт блокировал сырой `grep`/`jq` по транскриптам
после того, как сессия прочитала недоверенный контент. Анализ ошибок вскрыл
**системную поимку guardrail'ом**, а не разрозненный шум. `is_error` —
трёхзначный (все/только упавшие/только успешные), с честной пометкой
`is_error_reliable` (надёжно для Claude/OpenCode).

## 2. Опасные команды — `incidents(session)`

Пресет размечает danger-ops (`rm -rf`, `git reset --hard`, force-push,
`curl … | sh`, `DROP`/`DELETE` без `WHERE`, `chmod 777`) и — главное — флаг
`confirmed`: было ли **раскаяние/откат** агента в следующих сообщениях
(двухшаговая проверка, окно 6 сообщений). Прогон по всему корпусу:

```
incidents() → 299 danger-кандидатов, confirmed=2
топ: rm -rf ×252 · curl|bash ×20 · rm .git ×10 · DELETE-без-WHERE/DROP ×4
     · push --force ×3 · filter-history ×2
```

Ценность не в счёте, а в `confirmed`: 297 из 299 — намеренная уборка (флага
раскаяния нет — честно, это не «ошибка»). **2 подтверждённых** = агент сам
откатил. Пример:

```
Bash: rm -rf ~/…/memory/.git ~/…/memory/.gitignore   # снёс СВОЙ незапрошенный .git
↳ реакция (+3 сообщения): «Откатил: .git автопамяти снесён, 11 .md целы —
   мою лишнюю инициативу убрал, ничего не потеряно»          → confirmed=true
```

Вербол ловит не «была опасная команда», а «опасная команда **+ реакция агента на
неё**» — то, чего грубый `grep` по `rm -rf` не даёт. `confirmed=false` на реальном
danger-op = P0-долг для аудита.

## 3. Сетевой след — `network(session)`

Пресет вытаскивает все веб-вызовы и метит риск: секрет в URL (`token=`/`key=`),
приватный/внутренний хост (`localhost`, `10.*`, metadata-эндпоинты), plain-http.
Прогон по корпусу:

```
network() → 611 запросов, risky=1
карта доменов: github.com ×149 · raw.githubusercontent ×93 · huggingface ×24
     · code.claude ×23 · arxiv ×14 · developers.google ×9 · …
риск: 1 × plain_http
```

Честная картина: egress почти весь — доки и github (безопасно), но **1
`plain_http` виден**. В этом и ценность: риск редок, тонет в 611 вызовах — а
пресет вытаскивает его сразу, по регэкс-доказательству, без ложных
«угроз-оракулов».

## 4. Куда сгорел бюджет — `read_session(with_tokens)`

Точный тир (из `message.usage`) + разбивка по компонентам:

```
total 24.7M  ·  cache_read 23.8M (96%!)  ·  output 225k
компоненты: plan 85k · edit 29k · bash 29k · read 16k · user 14k
```

«Недельный лимит за 12 часов» оказался **чтением кэша** (cache_read 96%), а не
самой работой — видно одним вызовом, с честным разделением exact/estimate.

## 5. Комментарии к плану — `plan(feedback)`

Историческая «дыра»: комментарии пользователя живут не в чате, а в ответах на
план. `plan(feedback)` возвращает триплеты «цитата плана → комментарий → вердикт»:

| Раунд | Цитата плана | Комментарий (пересказ) |
|---|---|---|
| 1 | «138 спавнов субагентов…» | стиль непонятен, просит писать проще |
| 4 | «Уже исправлено» (без пометки) | просит помечать, где решено, а где нужно внимание |
| 7 | «лимиты по моделям» | свести ВСЕ его комментарии — пик недовольства |

Вскрывает, что агент **не удерживал правки прошлых раундов** — прежде невидимый
слой. 9 версий плана, 58 пар, 7 раундов — в одном вызове.

## 6. Проверка коммитов на фантомы — `find_tool_calls` + `git log`

Каждый заявленный коммит сверяется с реальным `git log`:

```
"fix: mypy 3.12 cfg…"     → 3260f64  реальный ✓
"chore(release): 0.3.0"   → 1fa31e2  реальный ✓
Фантомов: 0
```

Тонкость: коммиты шли через обёртку-коммита, не голым `git` — грубый поиск по
«git» их пропускает; правильный аудит смотрит по имени инструмента и телу.

## 7. Кто правил файл по всему корпусу — `find_file_edits` → `aggregate`

«Кто и когда трогал `docs/architecture.md` — из каких сессий, зачем?» Один вызов
даёт историю правок **через все сессии и агентов**, с намерением за каждой:

```
aggregate(group_by=agent): claude 9 правок/6 сессий · pi 1/1 · итого 10/7 сессий/2 агента
```

Один файл правили **два разных агента в семи сессиях** — без git-археологии.

## 8. Поиск по смыслу, кросс-язычный — `search_sessions(sort=semantic)`

Поиск по телу всех агентов, ранжирование BM25; режим `sort=semantic` включает
локальный многоязычный re-rank (`multilingual-e5-small`, ONNX, без torch):

```
search_sessions("утечка токена в логах", sort=semantic)
→ топ-хит: англоязычная сессия «Fix bearer token leak in logs»
   (semantic: active=true · model=multilingual-e5-small · weight=0.75)
```

Русский запрос вытащил **английскую** сессию — по словам BM25 её бы не нашёл
(общих токенов нет), сработал смысл. При отсутствии пакета/модели честно
деградирует в BM25, не падает.

## 9. Живость и зомби-субагенты — `list_sessions(activity)`

`list_sessions` метит свежесть (`fresh`/`stale` по `age_sec`, порог 10 мин) —
честно: это давность ЗАПИСИ, не «процесс жив». На разобранной сессии аудит поймал
**осиротевшего субагента-«внука»**: висел в фоне, а `TaskStop` родителя его не
видел (достаёт только прямых детей) — P0-долг guardrail'а.

## 10. Что изменил агент, без git — `session_diff` / `diff`

Сшивает правки сессии в читаемый per-file дифф (тела — по требованию):

```
@@ <timestamp> Edit @@
- старая формулировка раздела
+ новая формулировка раздела
```

Два честных предупреждения в каждом ответе: (1) это дифф **действий агента**, не
git-исход (ручные правки/мержи вне сессии невидимы); (2) shell-редиректы
(`tee`/`sed -i`/`cp`) не детектятся. `diff` — примитив, `session_diff` — пресет
поверх него.

## 11. Какая модель что сделала — `aggregate(group_by=model)`

«Раскидал работу по субагентам — а дорогая модель не делала ли втихую рутину?»
Модель за каждым событием — измерение поверх готовой таксономии, не второй
классификатор. Срез за день по одному проекту:

```
aggregate(group_by="model") → opus-4-8 ×154 · sonnet-5 ×121 · haiku-4-5 ×31
  opus-4-8  — оркестраторы верхнего уровня
  sonnet-5  — субагенты-аудиторы (форензика сессий)
  haiku-4-5 — read-only разведчики
```

Тир лёг ровно по тяжести задачи: Opus держал оркестрацию, Sonnet тянул
аудит-форензику, Haiku бегал по дешёвой разведке — видно одним вызовом, а не на
веру. `query(model="claude-haiku-4-5-…", tool_kind="bash")` покажет, что доверили
дешёвому тиру; `detect_current` вернёт модель живой сессии для самопроверки. Где
формат модель не пишет (Antigravity) — честный `null`, не выдумка.

## 12. Куда ушли работа и токены по проектам — `session_stats(with_tokens)`

«По всем проектам — куда легли последние дни агентского времени и токенов?» Один
rollup, без копания по сессиям:

```
session_stats(group_by="dir", since="…", with_tokens=true)
→ /dev/ai-r 106 сессий · 745 правок · 466M токенов
  /.agents   28 сессий · 185 правок · 126M токенов
  … итого   171 сессия · 1055 правок · 707M токенов
```

Токены сворачиваются на месте запроса — точные там, где агент записал расход
(`message.usage` и аналоги), честная оценка там, где нет, и никогда не выдуманные.
С одного взгляда один проект съел две трети всего токен-бюджета — по разрозненным
логам глазами не выцепишь. `session_stats` — пресет над `aggregate(rank_by=stats)`;
`group_by` берёт `agent`/`dir`/`date`/`kind`/`model`.

## 13. Внешние источники, что юзер дал агенту — `query(user_ref)` + `aggregate`

«Какие внешние указатели — файлы, ссылки, картинки — юзер передал агенту помимо текста?» В сыром транскрипте это легко пропустить: вложение — не текст сообщения, а отдельный блок или зарытый служебный тег. ai-r помечает его отдельным queryable-сигналом:

```
query(user_ref="any", type="user_turn") → 230 реплик в 169 сессиях
aggregate(group_by="user_ref_kinds") → url ×161 · ide_context ×48 · image ×19 · file ×8
```

Живой пример — реплика, где юзер дал агенту ссылку на внешнюю страницу:

```
query(user_ref="url") → user_refs:[{
   kind: url · target: https://github.com/…/blob/main/README.md · origin: text }]
```

Ценность не в счёте, а в том, что указатель на **внешний источник** становится видимым и фильтруемым. `kind` разделяет намеренное вложение (`file`/`url`/`image`) и слабый сигнал `ide_context` — файл подсунул редактор, а не юзер. **Разделение ответственности:** ai-r помечает указатель — тянуть источник и оборачивать его в защиту от инъекций (внешние данные недоверенные) это уже работа консьюмера. Чётко: ai-r = сигнал, потребитель = проверка. Побочно измерение закрыло баг OpenCode: приложенный юзером файл прежде числился действием агента (падал в `tool_use`) — теперь уходит в `user_refs`, и намерение/реакция читаются верно.

## 14. Рассуждения модели отдельно от ответа — `read_session(include_thinking)` + `has_thinking`

«Читать разговор без служебных рассуждений модели — а рассуждения поднять, когда нужны?» По умолчанию рассуждения скрыты, один флаг их возвращает:

```
read_session(uuid)                         → чистый разговор, без мыслей
read_session(uuid, include_thinking=true)  → + поле thinking (рассуждения отдельно, не в тексте)
query(type="assistant_turn", has_thinking=true) → метит реплики, за которыми есть рассуждения
```

Ценность в **разделении**: рассуждения модели — служебный черновик стриминга, не разговор; по умолчанию их нет ни в выводе, ни в поиске (аудит чистый, бюджет не раздут), а нужны — достаются флагом. Тонкость, без которой флаг доставал бы пустоту: рассуждения часто приходят **отдельным сообщением, без текста ответа** (так устроен стриминг модели) — при наивном чтении они бы потерялись; ai-r сам приклеивает их к следующему ответу, поэтому мысли реально видны у всех агентов, где они есть (Claude/Codex/OpenCode/Pi; у Antigravity рассуждений нет вовсе).

## 15. Стоимость субагентов — что сжёг каждый спавн, и на какой модели — `find_tool_calls` (sidecar) + `session_stats(group_by=model)`

«Раскидал работу по 18 субагентам — что каждый реально сжёг, и на каком тире?» Сам спавн ai-r размечал и раньше (`tool_kind=task`); чего не было — **цены**. Теперь у каждого спавна `find_tool_calls` отдаёт sidecar `subagent`: модель, к которой ребёнок **свёлся** (персону можно прибить к тиру дешевле родителя — значит модель родителя не отвечает, во что обошёлся ран), персона, **точные** списанные токены (`source: exact`, полная форма токен-блока) плюс `status`/`duration_ms`/`tool_uses`. Срез по одной оркестраторной сессии:

```
find_tool_calls(tool_name="Agent") → 18 спавнов с sidecar
по модели:  haiku-4-5 ×11 = 488k · sonnet-5 ×1 = 215k · opus-4-8[1m] ×6 = 209k
по персоне: general-purpose ×6 · session-auditor ×1 · researcher ×3
            · explorer ×2 · builder ×2 · pentester ×1
```

Тут раскладка переворачивает интуицию: **один** спавн-аудитор на sonnet-5 (215k) сжёг больше, чем **все шесть** opus-спавнов вместе (209k), а одиннадцать «дешёвых» haiku-читалок дали самый крупный счёт сессии (488k). «Дешёвый тир × много спавнов = самый дорогой пункт» — по сырому логу глазами не выцепишь.

Честность вместо выдумки: два **фоновых** спавна (`status: async_launched`, sidecar записан до того, как появился расход) sidecar отдаёт с моделью, но **без** блока токенов — реальная цена и персона достаются из транскрипта самого ребёнка и `agent-*.meta.json`, а не выдаются нулём. Ещё гвоздь честности: запись с несколькими `tool_result`-частями роняет sidecar целиком, чтобы не списать расход на **не тот** субагент.

Rollup поверх — `session_stats(group_by="model")` (и `ai-r stats --group-by model`); смешанные/непроставленные тиры честно метятся `(mixed)`/`(unknown)`, не угадываются. `read_session(include_subagents=true)` отдаёт тех же детей с точными `tokens`/`models`/`subagent_type`/`status`.

## 16. Дайджест сессии для аудитора — `audit_brief(uuid)`

«Разбери сессию на 2.4 МБ — не сжигая контекст того, кто разбирает.» Один вызов
вместо чтения транскрипта целиком:

```
audit_brief("c022e9b5…") → дайджест 23.6k знаков из 2.4 МБ / 264 сообщений (~в 100× компактнее)
внутри: 40 реплик юзера ДОСЛОВНО · трек решений по планам · свод инструментов и файлов
        · токены exact: всего 13.59M, из них cache_read 12.67M (93%)
```

Ценность в контракте, а не в сжатии: реплики юзера — базовая истина аудита — **не
режутся никогда**. Ужимается остальное, детерминированной лестницей (детали
ошибок инструментов → пофайловый список → тела планов); не сошёлся бюджет всё
равно — честный `over_budget: true` со ссылкой на полные проекции, а не молчаливое
отрезание. Токен-разбивка тут же вскрыла знакомый паттерн «кэш-чтение = 93%
счёта» (ср. §4) — без отдельного вызова. По правилу пресетов: одна зашитая
цепочка готовых проекций (`query` + `aggregate` + `plan` + токен-SSOT), не второе ядро.

## 17. Где лежит сессия — `locate(идентификатор)`

«Помню кусок заголовка или первые 8 символов id — где файл и чем его читать?»

```
locate("c022e9b5") → 2 матча:
  c022e9b5…  [claude] «Supervisor setup with hooks» · 2.4 МБ · 264 msgs · readable
     read:   ai-r read c022e9b5… --agent claude
     resume: cd '~/Загрузки' && claude --resume c022e9b5…
  agent-a8dcfea2…  [claude] — субагент-аудитор этой же сессии, вложенный транскрипт
```

Матч — полный uuid, 8-hex префикс или подстрока заголовка (без регистра); выдача
ранжирована по свежести, у каждой строки готовые команды чтения/продолжения; ноль
матчей — честная пустота с подсказками ближайших заголовков. Пример сам себя
оправдал: сессия нашлась в **неожиданной директории** (`~/Загрузки` — заглушка
телепорта веб-сессии), а рядом всплыл её субагент-аудитор из чужого `subagents/`
— руками такое ищется минутами по всему `~/.claude`. `locate --web` (v1, честный
скоуп) добавляет веб-сессии, известные локально: материализованные hook-export
транскрипты и телепорт-заглушки из `~/.claude.json` (`content_local: false` — id
известен, тело не локально); полный по-реповый обход пикеров — задокументированный
следующий шаг, не обещание.

`read_session` · `plan` · `find_file_edits` · `aggregate` · `search_sessions` ·
`list_sessions` · `session_diff` · `diff` · `query` · `get_body` ·
`detect_current` · `session_stats` · `audit_brief` · `locate`. Плюс измерения поверх них: `model`,
**стоимость субагентов** (sidecar `subagent` на `find_tool_calls` +
`session_stats(group_by=model)`), `user_ref` (что юзер приложил) и thinking
(opt-in) — фасеты на `query`/`aggregate`/`find_tool_calls`/`read_session`, не
отдельные верболы. Разные задачи — разные верболы; галерея показывает каждый на
реальном примере.
