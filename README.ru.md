**Агент отчитался «готово». Проверить — нечем.**

`ai-r` читает историю сессии любого из пяти агентов для кода и даёт свежему
агенту холодно проверить то, на что `git` не ответит:

- не наврал ли, не наломал ли — сдержал ли слово, не запускал ли опасного
  (и откатил ли, если запустил), что реально изменил, во сколько обошёлся;
- почему именно так — под каким планом, с каким намерением и чьей рукой правка.

> На своём же корпусе — 1600+ сессий пяти агентов в 20+ проектах — так нашли
> 312 рискованных команд (`rm -rf`, `curl|sh`, `git push --force`): две агент
> спохватился и откатил сам, остальные 310 прошли молча — `git` их не покажет.

`git` показывает, **что** уехало в код; `ai-r` — можно ли доверять тому, **как**
агент к этому пришёл. Только чтение: без LLM-вызовов и без сети.

## Быстрый пример — агент спрашивает историю

Главный режим — **MCP**: агент (Claude, Codex, …) зовёт `ai-r` напрямую и
спрашивает про историю обычным языком. Например — вытащить план, на котором
остановился прошлый агент, отбросив черновики:

```
Покажи план из прошлой сессии — только финальный, без промежуточных ревизий.
→ plan(session=…, kind="final")  →  get_body(id, shallow=true)

  план:            «Мигрировать auth на JWT: 1) вынести проверку…»
  dropped_drafts:  2   ← два черновика агент отбросил по дороге
  сессия:          a3f… (claude)
```

Быстрая атрибуция правок — одна команда в терминале, сразу по всем агентам:

```bash
ai-r find-file-edits auth.py --since 2026-06-01
```
```
2026-06-03  codex   auth.py  «добавь refresh-токен»            edit
2026-06-07  claude  auth.py  «вынеси проверку в middleware»    edit
```

## Что болит

- «Готово, сделал X по плану Y» — а проверить нечем: план агент держит в одном
  виде, правки в другом.
- Сменили агента посреди задачи — потеряли нить. Спросить «что *другой* агент
  уже пробовал?» негде.
- Всплыла правка в файле — непонятно, **какой** агент её сделал и по какому
  запросу.

Причина одна: каждый агент пишет историю **по-своему** — Claude и Codex в
JSONL, OpenCode в SQLite, Antigravity в «brain»-директориях, Pi в JSONL по
проектам. Пять форматов, пять раскладок — вместе они не сводятся.

## Обещание

`ai-r` сводит все пять в **один read-only интерфейс**. Наведите любого агента —
или скрипт, или себя — на любую сессию, неважно, какой инструмент её записал.
Одна форма запроса для каждого агента; различия форматов нормализуются внутри
парсеров.

Даже с **одним** агентом это работает: аудируешь свою же историю Claude (или
Codex…). Пять форматов — чтобы при смене инструмента история не рвалась, а не
требование иметь все пять.

### Как источник для RAG

В связке «LLM + внешний источник данных» (RAG) `ai-r` — **источник**, а точнее
слой извлечения над сессиями агентов. На запрос он возвращает не кусок лога, а
разобранные сущности: план, намерение, авторство правки — со ссылкой на тело,
которое агент дотянет, если понадобится.

Он **не заменяет** твой RAG по коду и документации, а добавляет источник, до
которого тот не дотягивается. Из чего обычно извлекают: документация, история
коммитов, Stack Overflow, внутренние вики, кодовые базы, баг-репорты. Сессий
агентов в этом списке нет — хотя только там записано, **почему** правка вообще
случилась.

Извлекает BM25 (поиск по словам с ранжированием), опционально — семантическая
пересортировка. Ни векторной базы, ни второго LLM: всё локально, результаты
воспроизводимы. BM25 здесь не упрощение — GitHub перечисляет его наравне с
векторными ретриверами: [«Common retrievers include sparse methods like BM25 and
dense vector retrievers using neural
networks»](https://github.com/resources/articles/software-development-with-retrieval-augmentation-generation-rag).

## Ключевые возможности

Каждый пункт — вопрос доверия с первого экрана и верб, что на него отвечает:

- **Сдержал ли слово — план vs факт.** Достаёт финальный план (отдельно от
  выброшенных `dropped_drafts`) и сверяет с тем, что реально уехало в правки —
  ловит «сделал X по плану Y», где Y уже не тот. (`plan`, `session_diff`)
- **Не запускал ли опасного — и откатил ли.** Помечает рискованные команды
  (`rm -rf`, `curl|sh`, `git push --force`) и по следующим репликам видит:
  спохватился и откатил — или прошло молча. (`incidents`, `query tool_kind=bash`)
- **Что реально изменил и чьей рукой.** Любая правка/вызов → агент, что её
  сделал, плюс запустивший её запрос; включая правки через shell (`> файл` у
  codex), которые обычный дифф пропускает. (`find-file-edits`, `find-tool-calls`)
- **Во сколько обошёлся.** Токены и стоимость сессии — точные там, где формат
  записал расход, честная оценка там, где нет, никогда не выдуманные.
  (`session_stats with_tokens`, `aggregate group_by=model`)
- **Почему именно так.** Намерение за правкой (запрос *перед* ней), под каким
  планом, какой моделью — «почему», а не только «что». (`query with_intent`)
- **Маленький ответ, тело по требованию.** Запись несёт ссылку на содержимое
  (хэш + длина); полный текст — отдельным запросом. Читалка, не охранник:
  только чтение, ничего не запускает и не пишет в историю агента.

## Как ai-r это узнаёт

Детерминированно, без второго LLM-домысла — и честно про границы:

- **опасная команда** — паттерн по строке вызова (`rm -rf`, `curl|sh`,
  `git push --force`, …). Обфусцированное (`exec(input())`) шаблон не ловит — это
  заявленная граница, не тихий провал.
- **откат** — помечается «подтверждён» ТОЛЬКО если рядом (в окне следующих
  реплик) есть маркер сожаления/апологии агента (сам маркер — двуязычный ру/en
  паттерн, не LLM-оценка настроения). Нет маркера → остаётся неподтверждённым
  кандидатом: `ai-r` **не додумывает молчаливый откат**, а честно говорит «не
  подтверждён».
- **соврал про план** — `ai-r` не решает за тебя. Кладёт рядом сущность-план и
  реально реконструированные правки сессии (`session_diff`) — расхождение видно
  тебе или проверяющему агенту. Это сведение evidence, а не семантический вердикт.

Ноль LLM-вызовов, только чтение — цифры воспроизводимы, а «подтверждён» никогда не
угадывается.

## Зачем это

- **Аудит сессий свежим взглядом.** Новый агент с пустым контекстом холодно
  проверяет прошлые сессии по трём осям: выполнены ли обещания и требования;
  логичны ли решения и их качество; насколько глубоко изучен вопрос — что агент
  упустил. Так ловятся агенты, которые задачу сделали, **но при планировании
  ввели в заблуждение** — в живом чате это проходит мимо и уводит в неверные
  решения.
- **Продолжить, когда кончился контекст — без потери деталей.** `/compact`
  затирает подробности. Вместо этого открой новую сессию: она прочитает **логи**
  предыдущей и продолжит с её выводов, не сжигая контекст заново на то, что уже
  изучено. Исходная сессия остаётся целой — для аудита и поиска. Новая сессия
  может быть в **любом** агенте: история сводится независимо от инструмента.
- **Питает твою систему памяти.** Ведёшь память и саммари по методу Карпатого
  или своему? `ai-r` даёт для AI-чатов то же, что ты делаешь с перепиской, —
  разобранные сущности, из которых строишь постоянную память важных деталей.
- **Вспомнить, что и зачем делали.** Зачем правили этот файл? Почему завели это
  правило? Находишь сессию, где файл менялся, и читаешь запрос *перед* правкой.

## Чем отличается от инструментов поиска по сессиям

Несколько кросс-агентных инструментов теперь читают историю больше чем одного
агента (`jazzyalex/agent-sessions`, `Dicklesworthstone/coding_agent_session_search`,
`hacktivist123/agent-session-resume`). Почти все — про **поиск и таймлайн**:
найти *сессию*, пролистать историю.

`ai-r` идёт глубже: достаёт **план, намерение и авторство готовыми сущностями**,
на которых ты строишь память. Поиск находит текст — `ai-r` отвечает на
**почему**. Технически поисковик тоже мог бы выкопать план из текста сессии, но
не отдаёт его разобранным в единую нормализованную форму — у `ai-r` это главная
поверхность.

| Возможность | Одноагентные вьюеры | Кросс-агентные поисковики | `ai-r` |
|---|---|---|---|
| Читает логи >1 агента | Нет | Да | Да — Claude, Codex, OpenCode, Antigravity, Pi |
| Программная поверхность | В основном GUI/TUI | В основном TUI/CLI/приложение | **MCP + CLI + Python SDK** |
| Атрибуция (правка/команда → агент + намерение) | — | Частично | Да — `find-file-edits` / `find-tool-calls` |
| Реплей аудита (реконструкция правок сессии, без git) | — | Редко | Да — `session_diff` |
| Извлечение плана (финал vs черновик, нормализовано) | — | — | Да — `plan` |
| Охват | Вьюер | Поиск / resume / память | **Read-only ядро извлечения** |

*Столбцы конкурентов отражают их публичные доки на 2026-07; где возможность
неясна — мы скорее занижаем, чем переобещаем.*

Мы намеренно **не** соревнуемся по числу агентов, скорости или богатству TUI.
Клин `ai-r` — извлечение «почему» и структурированных сущностей для машинного
потребления.

## Проверено на деле

`ai-r` уже читает историю собственной разработки — по всем пяти агентам. На нём
работают реальные инструменты (они живут отдельно, поверх его read-only API):

- **аудитор** — свежий агент холодно проверяет, что предыдущий на самом деле
  сделал и решил. Так поймали агентов, тихо приврав­ших про план.
- **суммаризатор** (`export rounds`) — рендерит сессию в готовый handoff-док.
- **ai-local-reader** — read-only скилл: аудит прошлых сессий с диска по всем
  агентам.

Эти инструменты — на стороне workflow, вне этого репозитория. Сам `ai-r` только
читает и возвращает данные.

## Поверхности

`ai-r` даёт одну и ту же силу чтения тремя способами:

- **MCP-сервер** (`ai-r-mcp`) — 15 инструментов через JSON-RPC, так что
  любой MCP-агент дёргает его напрямую (рекомендуется). По умолчанию — **stdio**;
  опционально — **общий http-сервер** (один тёплый процесс на всех агентов
  вместо роя stdio по процессу на агента), см. дополнение `http` в «Быстром
  старте». Регистрация — см.
  [docs/mcp-registration.md](./docs/mcp-registration.md).
- **CLI** (`ai-r`) — субкоманды для скриптов и ручного использования
  (`list` / `read` / `search` / `find-file-edits` / `find-tool-calls` /
  `file-frequency` / `detect-agent` / `export rounds`). Операторы поиска —
  [docs/search-operators.md](./docs/search-operators.md).
- **Python SDK** (`from ai_r.parsers import ...`) — парсеры, типизированные
  модели `Session`/сообщений и событийные вербы, чтобы строить свои
  инструменты.

### Словарь методов

Полный словарь публичных вербов и пресетов (сигнатуры, параметры, поведение) — в отдельном файле:

### Событийное ядро

Вербы выше — новые: одно **событийное ядро** заменяет кучу разовых
инструментов. Каждый парсер читает логи одного агента и выдаёт типизированные
модели, которые нормализуются в единый агент-нейтральный поток —
`user_turn` / `assistant_turn` / `tool_call(...)` / `plan_event`. Небольшой
набор вербов фильтрует, агрегирует и diff-ит этот поток; различия агентов
(`ExitPlanMode` против `update_plan` против `implementation_plan.md`) скрыты
внутри парсеров — вызывающий видит одну форму.

Честно про границу: это **только извлечение сущностей** — реплики, вызовы
инструментов, планы, намерения, реакции. Это **не** граф и **не** хранилище
памяти. Что делать дальше (граф знаний, Obsidian, постоянная память) — уже на
твоей стороне, вне этого репозитория. Полную слоёную схему и список
MCP-инструментов см. в [docs/architecture.md](./docs/architecture.md).

## Быстрый старт

**Попробовать без установки** — если есть [uv](https://docs.astral.sh/uv/):

```bash
uvx --from agent-session-reader ai-r list          # CLI: показать сессии
uvx --from agent-session-reader ai-r-mcp           # MCP-сервер (stdio)
```

Ничего не ставится в систему: `uvx` скачивает пакет во временный кэш и
запускает. Годится, чтобы посмотреть на свои сессии прямо сейчас или прописать
`ai-r-mcp` в MCP-конфиг агента вручную.

**Полная установка (1 команда)** — она же чинит конфиги за вас:

```bash
git clone https://github.com/pro-target/ai-r.git ~/dev/ai-r
cd ~/dev/ai-r && bash install.sh
```

Требования: Python 3.11+ с `venv` или `pip`, и `jq` (для автопатча MCP-конфигов
Claude и Antigravity — остальным `jq` не нужен).

Установщик создаёт venv, ставит runtime-пакет, патчит MCP-конфиги для
**Claude**, **Codex**, **OpenCode**, **Antigravity** (где конфиги существуют),
ставит CLI-скилл **Pi** и прогоняет смоук-тесты. Именно этого автопатча `uvx`
не делает — там конфиги правите сами.

Необязательное дополнение — `tokens`: `AI_R_EXTRAS=tokens bash install.sh`
(или `pip install "ai-r[tokens]"`) добавляет
[tiktoken](https://github.com/openai/tiktoken) — более точные **оценки**
токенов для сессий, чей формат не хранит точных чисел расхода. Полностью
опционально: без него точные числа по-прежнему берутся прямо из файлов сессий
(где записаны), а оценка деградирует до грубой эвристики «символы/4» с честной
пометкой `estimate` — никогда не падение.

**Необязательное дополнение — `semantic`** (`AI_R_EXTRAS=semantic bash install.sh`
или `pip install "ai-r[semantic]"` + разовое скачивание модели, которое
установщик делает сам): включает `sort="semantic"` в текстовом поиске (`query`,
`search_sessions`) — топ-50 кандидатов от BM25 пересортируются по **смыслу**.

- **Модель.** Локальная многоязычная модель эмбеддингов (embeddings)
  [intfloat/multilingual-e5-small](https://huggingface.co/intfloat/multilingual-e5-small)
  (int8 ONNX, ~118 МБ, MIT) — напрямую через
  [onnxruntime](https://onnxruntime.ai) + [tokenizers](https://github.com/huggingface/tokenizers) + [numpy](https://numpy.org),
  без torch и без постоянного индекса. Выбрана за сильный межъязыковой поиск
  (русский запрос находит английскую сессию и наоборот) при малом размере.
- **Как считается балл.** BM25 отбирает 50 лучших совпадений по словам (это
  бюджет затрат, а не отсечка по качеству — порога схожести намеренно НЕТ: у
  этого семейства моделей даже несвязанные тексты получают ≈0.7). Внутри пула
  итоговый балл = **75 % смысл + 25 % совпадение слов** — смысл главнее, а
  словесная доля не даёт утопить точное вхождение термина и разбивает ничьи.
- **Fail-soft.** Без пакетов или файлов модели `sort="semantic"` честно
  откатывается к порядку BM25 с объяснением
  (`semantic: {active: false, reason, fallback: "bm25"}`) — никогда не падение.

**Необязательное дополнение — `http`** (`AI_R_EXTRAS=http bash install.sh` или
`pip install "ai-r[http]"`): добавляет [uvicorn](https://www.uvicorn.org) и
включает **общий streamable-http транспорт** (требует `mcp>=1.9.0`).

- **Зачем.** По умолчанию каждый агент поднимает свой `ai-r-mcp` по stdio — под
  многоагентным веером это N процессов, каждый со своим холодным кэшем,
  перечитывающих корпус (измеренная причина исчерпания RAM). С
  `AI_R_MCP_TRANSPORT=http` вместо роя — **один тёплый сервер** на localhost
  (по умолчанию `127.0.0.1:8756`), общий для всех; systemd-юниты в
  `packaging/systemd/` дают socket-activation с само-выходом по простою.
- **Безопасность (fail-closed).** Bind — только loopback. Браузерные атаки
  (DNS-rebinding) отсекает Origin/Host-allowlist SDK (всегда включён для
  loopback). Удалённый доступ требует `AI_R_MCP_ALLOW_REMOTE=1` **и** токена
  `AI_R_HTTP_TOKEN` — без токена не стартует (транскрипты содержат секреты). На
  loopback токен опционален (защита от другого локального юзера общей машины);
  клиент шлёт заголовок `Authorization: Bearer <token>`.
- **Ручки (env):**
  - `AI_R_MCP_PORT` — порт (по умолчанию `8756`).
  - `AI_R_MCP_IDLE_SEC` — порог само-выхода по простою.
  - `AI_R_MCP_HOST` / `AI_R_MCP_ALLOW_REMOTE` — хост bind / разрешить не-loopback.
  - `AI_R_HTTP_TOKEN` — bearer-токен (обязателен для удалённого bind).
  - `AI_R_HAYSTACK_CACHE_MAX` — потолок кэша поиска по числу записей.
  - `AI_R_HAYSTACK_CACHE_CHARS_MAX` — по суммарному объёму (предохранитель RSS
    долгоживущего сервера).

Оба дополнения полностью опциональны: без них stdio-режим и порядок BM25
работают как раньше.

## Границы: читалка, не охранник

- **Только чтение.** Никогда не запускает код агента и не пишет в его историю —
  читает и возвращает.
- **Ни графа, ни памяти.** Достаёт сущности (реплики, вызовы, планы,
  намерения). Строить из них граф знаний или память — твоя задача, не его.
- **Не контроль доступа — кроме http-транспорта.** Кто дотянулся до CLI, MCP по
  stdio или пакета — читает любую сессию: это тот же локальный юзер, проверка
  прав перед парсерами ничего бы не стерегла. Исключение — общий http-транспорт:
  он доступен по сокету, поэтому несёт Origin-allowlist и опциональный
  bearer-токен (обязательный для удалённого bind, см. дополнение `http` выше).
  Всё равно держи данные там, куда чужие локальные процессы не дотянутся.
- **Содержимое сессий — данные, не команды.** Кто читает (аудитор,
  суммаризатор), обязан относиться к тексту сессии как к данным, а не
  инструкциям. См. [Безопасность](docs/security.md).
