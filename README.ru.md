> `git` показывает, **что** попало в код. `ai-r` показывает, **почему**: какой
> агент это сделал, под каким планом — и не выронил ли он тихо тот план, на
> котором сам же остановился. Read-only, по всем пяти агентам для кода, один
> интерфейс.

Агент отчитался: «сделал X по плану Y». Проверить нечем. План — в одном
формате, правки — в другом. А если над задачей работали два агента, их истории
вообще не сводятся: каждый пишет по-своему и в своём месте. `ai-r` читает
историю сессий агента и достаёт из неё намерение, план и авторство за правкой.

## Быстрый пример — агент спрашивает историю

Главный режим — **MCP**: агент (Claude, Codex, …) зовёт `ai-r` напрямую и
спрашивает про историю обычным языком. Например — вытащить план, на котором
остановился прошлый агент, отбросив черновики:

```
Покажи план из прошлой сессии — только финальный, без промежуточных ревизий.
→ ai-r: plan(session=…, kind="final")  →  get_body(id, shallow=true)
        возвращает финальную задачу + список dropped_drafts
```

Быстрая атрибуция правок — одна команда в терминале, сразу по всем агентам:

```bash
# кто правил этот файл и когда — кросс-агентно, опционально за период
ai-r find-file-edits auth.py --since 2026-06-01
```

## Что болит

- «Готово, сделал X по плану Y» — а проверить нечем: план агент держит в одном
  виде, правки в другом.
- Сменили агента посреди задачи — потеряли нить. Спросить «что *другой* агент
  уже пробовал?» негде.
- Всплыла правка в файле — непонятно, **какой** агент её сделал и по какому
  запросу.

Причина одна: каждый агент пишет историю **по-своему** — Claude и Codex в
JSONL, OpenCode в SQLite, Antigravity в «brain»-директориях, Pi в JSONL по
проектам. Пять форматов, пять раскладок — вместе они не сводятся.

## Обещание

`ai-r` сводит все пять в **один read-only интерфейс**. Наведите любого агента —
или скрипт, или себя — на любую сессию, неважно, какой инструмент её записал.
Одна форма запроса для каждого агента; различия форматов нормализуются внутри
парсеров.

## Ключевые возможности

- **«Почему?», не только «Что?».** Извлекает план, намерение и авторство за
  правкой — не только текст диффа. `git diff` говорит, *что* изменилось;
  `ai-r` — под каким планом и по чьему запросу.
- **Финальный план, а не черновики.** `ai-r` достаёт план, на котором агент
  *остановился*, и отдельно показывает, что он по дороге выбросил
  (`dropped_drafts`) — по Claude / Codex / Antigravity, где сигналы плана у
  каждого свои.
- **Кросс-агентная атрибуция.** Любая правка файла или вызов инструмента →
  агент, что её сделал, плюс запустивший её запрос (`find-file-edits` /
  `find-tool-calls`).
- **Маленький ответ, тело по требованию.** Записи несут ссылку на содержимое
  (хэш + длину), а полный текст правки берётся отдельным запросом — ответ не
  раздувается.
- **Работает через MCP (13 инструментов).** Агент зовёт `ai-r` напрямую
  обычным языком; те же данные доступны из терминала (CLI) и из кода (Python
  SDK).
- **Читалка, не охранник.** Достаёт сущности; граф знаний и память строишь ты
  (или твой инструмент). Только чтение: ничего не запускает и не пишет в
  историю агента.

## Зачем это

- **Аудит сессий свежим взглядом.** Новый агент с пустым контекстом холодно
  проверяет прошлые сессии по трём осям: выполнены ли обещания и требования;
  логичны ли решения и их качество; насколько глубоко изучен вопрос — что агент
  упустил. На реальном прогоне так за неделю разобрали **271 диалог** и нашли
  агентов, которые задачу сделали, **но при планировании ввели в заблуждение** —
  в живом чате это проходит мимо и уводит в неверные решения.
- **Продолжить, когда кончился контекст — без потери деталей.** `/compact`
  затирает подробности. Вместо этого открой новую сессию: она прочитает **логи**
  предыдущей и продолжит с её выводов, не сжигая контекст заново на то, что уже
  изучено. Исходная сессия остаётся целой — для аудита и поиска. Новая сессия
  может быть в **любом** агенте: история сводится независимо от инструмента.
- **Питает твою систему памяти.** Ведёшь память и саммари по методу Карпатого
  или своему? `ai-r` даёт для AI-чатов то же, что ты делаешь с перепиской, —
  разобранные сущности, из которых строишь постоянную память важных деталей.
- **Вспомнить, что и зачем делали.** Зачем правили этот файл? Почему завели это
  правило? Находишь сессию, где файл менялся, и читаешь запрос *перед* правкой.

## Чем отличается от инструментов поиска сессий

Появилась горстка кросс-агентных инструментов, читающих историю нескольких
агентов (`jazzyalex/agent-sessions`,
`Dicklesworthstone/coding_agent_session_search`, `hacktivist123/agent-session-resume`).
Почти все они — про **поиск и таймлайн**: найти *сессию*, пролистать историю.

`ai-r` идёт глубже: он извлекает **план, намерение и авторство как готовые
сущности**, на которых ты строишь память. Поиск находит текст — `ai-r`
отвечает, **почему**. Технически поисковый инструмент тоже мог бы выудить план
из текста сессии, но он не отдаёт его наружу в разобранном, едином виде — у
`ai-r` это главная поверхность.

| Возможность | Вьюеры одного агента | Кросс-агентные search-тулы | `ai-r` |
|---|---|---|---|
| Читает логи >1 агента | Нет | Да | Да — Claude, Codex, OpenCode, Antigravity, Pi |
| Программная поверхность | В основном GUI/TUI | В основном TUI/CLI/app | **MCP + CLI + Python SDK** |
| Атрибуция (правка/команда → агент + intent) | — | Частично | Да — `find-file-edits` / `find-tool-calls` |
| Аудит-реплей (реконструкция изменений сессии, без git) | — | Редко | Да — `session_diff` |
| Извлечение плана (final vs draft, нормализовано) | — | — | Да — `plan` |
| Скоуп | Вьюер | Поиск / резюм / память | **Read-only ядро извлечения** |

*Столбцы конкурентов — по их публичным докам на 2026-07; где возможности
неясны, мы скорее занижаем, чем переоцениваем.*

Мы сознательно **не** соревнуемся по охвату агентов, скорости или богатству
TUI. Клин `ai-r` — извлечение «почему» и структурные сущности для
машинного потребления.

## Проверено в деле

`ai-r` уже читает собственную историю разработки — по всем пяти агентам. На нём
держатся реальные инструменты (живут отдельно, поверх его read-only API):

- **аудитор** — свежий агент холодно проверяет, что предыдущий реально сделал и
  решил. Так ловили агентов, которые тихо привирали про план.
- **суммаризатор** (`export rounds`) — рендерит сессию в готовый
  документ-передачу (handoff).
- **ai-local-reader** — read-only скилл: аудит прошлых сессий с диска по всем
  агентам.

Эти инструменты — на стороне рабочего процесса, вне этого репозитория. Сам
`ai-r` только читает и отдаёт данные.

## Поверхности

`ai-r` даёт одну и ту же силу чтения тремя способами:

- **MCP-сервер** (`ai-r-mcp`) — 13 инструментов через stdio JSON-RPC, так что
  любой MCP-агент дёргает его напрямую (рекомендуется). Регистрация — см.
  [docs/mcp-registration.md](./docs/mcp-registration.md).
- **CLI** (`ai-r`) — субкоманды для скриптов и ручного использования
  (`list` / `read` / `search` / `find-file-edits` / `find-tool-calls` /
  `file-frequency` / `detect-agent` / `export rounds`). Операторы поиска —
  [docs/search-operators.md](./docs/search-operators.md).
- **Python SDK** (`from ai_r.parsers import ...`) — парсеры, типизированные
  модели `Session`/сообщений и событийные вербы, чтобы строить свои
  инструменты.

## Verbs

| verb | назначение | параметры |
|---|---|---|
| `query` | фильтр/поиск событий сессий — выдаваемый `text` = **превью** ~160 символов (обрезка после редакции секретов; реальная обрезка помечается хвостовым `…` + `text_truncated: true`), полное тело по требованию через `get_body`; `with_intent=True` → на каждое событие top-level `intent` (та же `previous_user_intent`, что у legacy); событие `tool_call` несёт ref `is_error` (исход вызова), когда его результат коррелируется, плюс обёртко-осознанные `tool_kind`/`tool_resolved` (см. *Tool-вызовы под обёртками* и *Границы вывода и исход* ниже) | type, agent, session (uuid \| **список uuid** — объединение событий этих сессий; пустой список / не-строка падают громко), since, until, file, tool (матчит и resolved-имена), tool_kind(edit\|write\|read\|bash\|task\|skill\|mcp\|web\|other), text, sort(relevance\|date\|semantic), relative_to+direction(prev\|next)+n(1\|all), step_type, limit, with_intent, noise(include\|exclude\|only), project_dir, redact; kind/parent/group — заглушки (Phase 3) |
| `plan` | нормализованные plan-атомы сессии (final vs drafts, группировка по задачам); по умолчанию инлайнится полный текст **финального** плана (`body` + `body_source`) и под `feedback` возвращаются все пары «цитата плана → комментарий пользователя» — с `version`/`plan_version`, якорями `section` и номерами `round` (см. *Итерации планов* ниже) | session, kind(draft\|final\|completed_major), group=task, agent, redact, bodies(final\|none), feedback(true\|false), rounds(all\|last) |
| `get_body` | тело on-demand по id события/плана; plan-feedback-ссылка `"<session>:pf<N>"` резолвится в ПОЛНЫЙ сырой ответ на план (тип `plan_feedback`); возвращаемое тело/текст ограничено `max_chars` (по умолчанию 500k) → превышение режется с маркером и помечается `body_truncated` | id, shallow, max_chars, redact |
| `aggregate` | rollup над rows (query/find_file_edits/session-inventory) → `{groups, totals}`; `rank_by=stats` даёт session_stats-порядок (sessions→edits→label), `kind_split=True` добавляет `kind_split_available`/`note`; метрика `tokens` сворачивает per-row токен-блоки в `{input, output, reasoning, cache_read, cache_write, total, exact, estimated, unknown}` (см. *Расход токенов* ниже) | rows, group_by(field\|callable), metrics ⊆ count\|sessions\|edits\|intents\|agents\|messages\|files\|tokens, rank_by(default\|stats), kind_split |
| `diff` | стичинг edit-rows в per-file unified diff (тела on-demand через message_index; `intent` берётся из row при `query(with_intent)`) → `{files:[{file,edits,diff,hunks}], count, caveats}` | rows, per_file=True, format=unified, redact |
| `detect_current` | runtime-identity (env/fs, вне session-query) → `{session_id, agent, candidates[], verified, self}` | agent (hint) |

## Аудит-пресеты: `incidents` и `network`

Оба следуют правилу пресетов — один вызов = зашитая цепочка базовых вербов с детерминированным алгоритмом внутри, никогда не второе ядро. Ноль LLM, ноль угадывания: каждый вердикт — попадание словаря/регэкспа по свидетельству транскрипта либо честный `null`.

- **`incidents` (F4.1)** — «где агент запустил что-то разрушительное — и извинился ли потом?». ОДИН скан `query(type="tool_call", tool_kind="bash")` даёт кандидатов; детерминированный **словарь опасности** (19 паттернов по категориям `fs`/`git`/`db`/`net`, собран из публичных guardrail-наборов и откалиброван на реальной истории хоста — Bash-`description` сам по себе никогда не срабатывает, `--force-with-lease` — не force-push) отбирает опасные команды; двуязычный (ru+en) **словарь сожаления** сканирует следующие `reaction_window` сообщений (по умолчанию 6) на извинение/откат — двухшаговая проверка за флагом `confirmed`, никогда не выводится косвенно. Каждая запись несёт `id` события query (контекст on-demand через `relative_to`/`read_session`), `patterns` + `categories`, обрезанный по символам фрагмент `command` с центром на попадании, tri-state `is_error` (`null`, когда формат агента не несёт коррелированного сигнала исхода) и `reaction` (метки маркеров + обрезанное превью; `null`, когда не подтверждено). `count`/`confirmed_count`/`by_pattern` всегда отражают ПОЛНОЕ множество совпадений независимо от `limit`; неизвестные значения `category`/`confirmed` падают громко; ноль инцидентов → `diagnostics` пустого результата. Задокументированная оговорка: словарь не отличает упоминание от выполнения — опасная строка внутри `echo` тоже может сматчиться. SSOT `ai_r.incidents`.
- **`network` (F4.3)** — «куда агент ходил в сеть — и насколько рискованно выглядели эти запросы?». ОДИН скан `query(type="tool_call", tool_kind="web")` даёт кандидатов (Claude `WebFetch`/`WebSearch`, OpenCode `webfetch`, Codex `web_search` — поднят из rollout-записей `web_search_call`, Gemini/Antigravity `web_fetch`/`google_web_search`; Pi веб-тула не пишет — честное отсутствие); цель запроса (`url`/`query`) извлекается из input самого вызова (нечего извлечь → честные `null`-поля и `kind: null`, никогда не угадывается по имени тула); детерминированный **словарь рисков** оценивает каждый запрос — `plain_http`, `credentials_in_url`, `secret_in_url`/`secret_in_query` (паттерны редакции F2.1 работают детектором — один словарь, два применения), `ip_literal_host`, `private_or_local_host`, `punycode_host`. Каждая запись несёт `id` события query, производный `kind` (`fetch`/`search`), обрезанные по символам `url`/`query` (обрезка применяется ПОСЛЕ редакции полной строки, поэтому секрет, разрезанный границей, никогда не утекает частично), `domain`, `risks` и tri-state `is_error`. `count`/`risky_count`/`by_domain`/`by_risk` всегда отражают ПОЛНОЕ множество совпадений независимо от `limit`; неизвестные значения `kind`/`risk` падают громко; `domain` матчится по равенству-или-поддомену; ноль запросов → `diagnostics` пустого результата. **Граница: сетевой доступ через MCP (браузерные MCP-тулы) остаётся под `tool_kind="mcp"` и никогда не угадывается в аудит — записанные в транскрипте имя+input — единственный сигнал, который есть у ai-r.** SSOT `ai_r.network`.

## Legacy-тулы: пресеты над вербами (Phase 3b)

Phase 3b обогатила вербы, чтобы старые тулы стали тонкими пресетами **с byte-identical выходом, доказанным на РЕАЛЬНЫХ данных** (frozen snapshot `~/.claude`, чтобы живой vault не мутировал в середине прогона — это давало ложные mismatch'и). Legacy-сьюты (`test_session_stats`/`test_session_diff`) зелёные — вторая половина доказательства совместимости.

**Переведены на вербы (byte-parity доказана):**

| тул | пресет над вербом | доказательство |
|---|---|---|
| `session_stats` | `aggregate(rank_by=stats, kind_split=True)` над per-session inventory-rows | 8/8 (group_by∈agent\|dir\|date\|kind × top∈8\|0) EQUAL на snapshot; ключ — `rank_by=stats` воспроизводит sessions-first ранк, `kind_split` даёт `kind_split_available`/`note` |
| `session_diff` (≠codex) | `diff(query(edit\|write, with_intent=True))` | 12/12 реальных Claude-сессий EQUAL; ключ — `with_intent` возвращает `intent`, единый chronological stream даёт тот же file-order, фильтр edit\|write исключает `Read` (иначе лишние файлы) |

**Codex — исключение в `session_diff`:** codex пишет файлы через shell-exec, target восстанавливается сканом командной строки, которого событийный поток НЕ делает → shell-redirect-правки исчезли бы из `query`-fold. Поэтому codex-ветка `session_diff` сохраняет legacy `_scan_session` (byte-parity для всех агентов).

**Остаются отдельными (обоснованно):**

| тул | почему НЕ пресет |
|---|---|
| `find_file_edits` / `find_tool_calls` | запись несёт `session_title`/`session_date`/`assistant`/`input`, которых НЕТ в событии `query`; `find_tool_calls` дополнительно несёт per-record `is_error` (коррелированный исход tool-вызова) и `output` (коррелированное содержимое tool-result, с char-cap); воспроизвести их = заново читать сессию (не *тонкий* пресет, а второй парс поверх событий — строго медленнее) + потеря codex shell-redirect-правок. `intent` теперь воспроизводим (`with_intent`), но остальных полей — нет. SSOT богатой edit/tool-записи |
| `search_sessions` | session-гранулярный + BM25-сниппеты сессий; `query` event-гранулярный (turn/tool) → чистого 1:1 нет |
| `detect-agent`/`detect-session` (CLI) | CLI печатает `source` агента и 6 режимов вывода (list/first/strict/self/fingerprint/`--json`/`--count`) + WARN-строку; дикт `detect_current` этого не даёт |

## Plan-атом (нормализованный, различия агентов скрыты)

`Plan { id, session_id, agent, title, task_id, kind: draft\|final\|completed_major, path?, steps?, status?, refs[], sha256 }`. Тело/steps — on-demand через `get_body(id, shallow?)`. `shallow=True` → только final задачи, тела draft-черновиков отброшены (сценарий S6).

**Группировка по задачам = `task_id` (стабильный ключ):** для Claude это slug плана `plans/<slug>.md` (Write несёт path напрямую; `ExitPlanMode` без path наследует slug ближайшего предшествующего plan-Write в сессии; если slug'а ещё не было — fallback на нормализованный title). Для Antigravity — путь `implementation_plan.md`. Для Codex (файла нет) — нормализованный title (непрерывный ран `update_plan`). Ключ по slug'у, а НЕ по title, потому что title дрейфует внутри одной итерации-цепочки (декорации меняют заголовок) — на реальных данных это резало одну задачу на несколько. В группе последний plan_event по (ts, seq) = `final`, ранние = `draft`; строго более ранние задачи (ДРУГОЙ slug) = `completed_major`. Внутренняя таблица parser→сигнал (`ExitPlanMode`/`Write plans/*.md`/`update_plan`/`implementation_plan.md`) — деталь реализации, наружу невидима.

## Итерации планов (тело финала, feedback, версии, раунды)

По умолчанию `plan` возвращает всё, что нужно консьюмеру, чтобы проиграть итерацию согласования плана, не инлайня каждый черновик (замерено ≈×3.7 дешевле, чем «все тела»; F3.4):

- **Тело финала инлайн** — `final`-план несёт `body` + `body_source`. АВТОРИТЕТНЫЙ текст — план, отредактированный пользователем в ответе-одобрении (`"approval_edited_by_user"` — файл плана на диске может расходиться с тем, что реально одобрили), фолбэк — сигнал плана (`"plan_signal"`); честный `null` для планов из одних шагов (Codex). Черновики остаются ссылками (тела через `get_body`); `bodies="none"` возвращает reference-only атомы.
- **Пары `feedback`** — ВСЕ пары «цитата плана → комментарий пользователя», извлечённые из ответов пользователя на план, хронологически. Каждая пара несёт `plan_id` (точная ревизия, которой отвечали; корреляция по call id — **отклонённый `Write` план-файла** коррелируется точно так же, как вердикт `ExitPlanMode`), `verdict` (`rejected` | `stay_in_plan_mode`), `quote` (`null` для free-text-комментария), дословный `comment`, `ts` и `ref` (`"<session>:pf<N>"`), который `get_body` резолвит в ПОЛНЫЙ сырой ответ (тип `plan_feedback`). Технические сбои и голые отклонения без комментария отфильтрованы; `feedback=false` убирает список. Сигнал есть только у агентов с интерактивным флоу согласования плана (сегодня: Claude `ExitPlanMode`); остальные честно не дают ничего — никогда не выдумывается.
- **Версии, секции, раунды (v2)** — каждый plan-атом несёт `version`, его 1-based номер ревизии внутри группы задачи в хронологическом порядке `(ts, seq)` (черновики — `v1…vN-1`, финал — `vN`; нумерация начинается заново на задачу). Каждая feedback-пара несёт `plan_version` (номер отвеченной ревизии, `null` без корреляции по call-id) и `section` — заголовок секции плана, к которой якорится цитата: пользователь выделяет цитаты из ОТРЕНДЕРЕННОГО плана, поэтому и цитата, и каждая секция сырого markdown сравниваются через одну и ту же нормализацию со срезанием разметки; цитата, не сматчившая НИ ОДНУ секцию — или БОЛЬШЕ одной — получает честный `null`-якорь, никогда не ближайшую догадку. Пары группируются по `round` — 1-based номеру feedback-раунда внутри сессии — а `rounds="last"` оставляет только финальный раунд каждой сессии (`"all"` по умолчанию; всё прочее падает громко). Сами plan-атомы от `rounds` не зависят.

Редакция (F2.1) покрывает тела планов, цитаты, комментарии и сырые ответы. SSOT ядра: `ai_r.events.plan_feedback`.

## Tool-вызовы под обёртками (`tool_kind` / `tool_resolved`)

Каждый tool-вызов классифицируется с учётом обёрток (F3.1). Каждое событие `tool_call` (`query`) и каждая запись `find_tool_calls` несёт `tool_kind` — одно из `edit|write|read|bash|task|skill|mcp|web|other` — и, когда input обёртки называет реального исполнителя, `tool_resolved`: тип сабагента под spawn-обёрткой (Claude `Task`/`Agent` → `input.subagent_type`, OpenCode `task` → `subagent_type`, Codex `spawn_agent` → `agent_type`), имя скилла под Claude `Skill` (`input.skill`) / OpenCode `skill` (`input.name`) / `SlashCommand` (`input.command`, сведённый к голому токену команды), либо `"<server>:<tool>"` для имени Claude-стиля `mcp__<server>__<tool>`. Только честные per-agent сигналы: обёртка, чей input не несёт ключа с именем, не получает `tool_resolved` (никогда не угадывается); Codex/OpenCode/Pi пишут MCP-вызовы под голыми или склеенными подчёркиваниями именами без надёжного разделителя сервера, поэтому детекции `mcp` там нет. `query` принимает фасет `tool_kind` (точное совпадение, неизвестное значение падает громко), а фасет `tool` матчит и resolved-имена (`tool="commit"` находит SlashCommand, запустивший скилл `commit`); `tool_kind`/`tool_resolved` подняты в top-level поля события, поэтому `aggregate(group_by="tool_kind")` работает по query-rows напрямую. Обратная совместимость: `type` события сохраняет базовый подтип `tool_call(<sub>)` — счётчики и существующие фильтры не меняются; `tool_resolved` проходит emission-time редакцию F2.1. Kind `web` (плюс поднятый парсером сигнал Codex `web_search_call`) питает пресет `network`. SSOT `ai_r.events._common.resolve_tool` + `TOOL_KIND`.

## Исход сессии (`read_session.outcome`)

`read_session` несёт блок `outcome` (F2.3) — `{status, signals, user_verdict, markers, tool_results, tool_errors, error_rate, error_rate_reliable}` со `status ∈ success|failure|mixed|unknown`. Два честных сигнала, никогда не догадка: (1) **доля ошибок tool-вызовов** — доля tool-result'ов, которые сам агент пометил как сбой; реальный флаг источника есть только у Claude (`tool_result.is_error`) и OpenCode (`state.status == "error"`), поэтому у Codex/Pi/Antigravity `tool_errors`/`error_rate` — `null` (`error_rate_reliable: false`) — зеркалит `find_tool_calls.is_error_reliable`; (2) **словарь вердикта пользователя** — двуязычные (ru+en) маркеры успеха/провала, матчатся только по последним 3 *человеческим* репликам пользователя (самоотчётам ассистента никогда не верим; XML-обёртки / плейсхолдеры `[...]` / преамбулы `Caveat:` пропускаются). Таблица решений: негативный вердикт → `failure`; позитивный → `success` (`mixed`, когда доминируют ошибки); нейтральный + доминирующие ошибки → `failure`; иначе `unknown` (пустые `signals` ⇔ `unknown`). Пороги и словарь откалиброваны на реальной истории («доминируют» = `rate ≥ 0.5` при `≥ 4` результатах) — консервативно по замыслу. Блок содержит только строки авторства ai-r и метки словаря (никогда не сырой текст сессии), поэтому проход редакции ему не нужен. SSOT `src/ai_r/outcome.py`.

## Расход токенов (`with_tokens` + метрика `tokens`)

`session_stats(with_tokens=True)` читает расход токенов каждой сматченной сессии из собственных файлов агента **в момент запроса** (ничего фонового, без индекса) и добавляет свёрнутый блок `tokens` к каждой группе и к `totals`; `aggregate` принимает метрику `tokens`, сворачивающую per-row токен-блоки так же (F3.3). Блок — `{input, output, reasoning, cache_read, cache_write, total, exact, estimated, unknown}` — суммы равны `null`, когда ни один row не нёс поле (никогда не выдуманный 0), а счётчики происхождения всегда удовлетворяют `exact + estimated + unknown == rows`. По сессии числа **точные (exact)** там, где формат пишет usage (Claude per-call `message.usage`, стриминговые дубликаты дедуплицируются по `(message.id, requestId)`; Codex — последний кумулятивный `token_count`; OpenCode — `message.data.tokens`; Pi — `usage` на assistant-сообщение) — через per-parser `read_token_usage` (фича для всех, где есть сигнал); сессия без записанного сигнала (Antigravity или старые данные) получает **оценку (estimate)** по объёму транскрипта с меткой `source="estimate"`: токенизация **опциональной** зависимостью `tiktoken` (`pip install "ai-r[tokens]"` / `AI_R_EXTRAS=tokens bash install.sh`), если она установлена, иначе грубая эвристика chars/4 — деградация, никогда не падение; полное отсутствие сигнала остаётся честным `unknown`. Наружу идут только целые числа и метки, посчитанные ai-r (никакого сырого текста сессий), поэтому блок по построению вне поверхности редакции. Дефолт `with_tokens=false` сохраняет byte-identical исторический вывод. SSOT `ai_r.tokens`.

## Семантическая сортировка (`sort="semantic"`)

Поверхность текстового поиска (`query` с фасетом `text`, `search_sessions`) принимает `sort="semantic"` (F5.1, опциональный `ai-r[semantic]`): топ-50 кандидатов BM25 пере-ранжируются по *смыслу* **локальной** мультиязычной embedding-моделью (`intfloat/multilingual-e5-small`, int8 ONNX, напрямую через onnxruntime + tokenizers — без torch; кросс-язычно ru↔en, синонимы). Постоянного индекса нет: тексты эмбеддятся в момент запроса, ничего не сохраняется. Пул топ-50 — бюджет стоимости, а не отсечка качества: порога похожести НЕТ намеренно, результаты только пере-упорядочиваются, никогда не выбрасываются; внутри пула смешанный балл = **75 % смысл + 25 % совпадение слов** (оба min–max-нормированы) — смысл доминирует, словесная доля защищает точные совпадения терминов и разруливает ничьи; результаты за пределами пула сохраняют порядок BM25. Ответ несёт отчёт `semantic`: `active: true` (+ модель, число кандидатов, вес смеси) либо честную деградацию `active: false` + `reason` простыми словами + `fallback: "bm25"` — без опциональных зависимостей (`pip install "ai-r[semantic]"`) или файлов модели (`AI_R_EXTRAS=semantic bash install.sh` их скачивает; переопределение через `AI_R_SEMANTIC_MODEL_DIR`) порядок откатывается к чистому BM25, никогда не падение, а дефолтные сортировки (`relevance`/`date`) модуль вообще не трогают. Эмбеддинг видит СЫРОЙ текст, вывод остаётся редактированным; reference-by-default без изменений. SSOT `ai_r.semantic`.

## Фильтр шума (session-уровень)

Сессия — *шум*, когда это не верхнеуровневый диалог с человеком: сегодня это **порождённые сабагент-сессии (sidechain)** — `kind == "subagent"` или задан `parent_uuid` (SSOT критерия: `src/ai_r/parsers/_noise.py`). `query`, `list_sessions` и `search_sessions` принимают `noise ∈ {include, exclude, only}` (по умолчанию `include` — полная обратная совместимость): `exclude` оставляет только верхнеуровневые agent-сессии, `only` — только дерево сабагентов (аудит-вид). Фильтр применяется на уровне *сессии* до чтения сообщений (исключённая сессия ничего не стоит), композируется с остальными фильтрами по И (включая `list_sessions(kind=…)`), неизвестный режим падает громко (`invalid_argument`). В `query` он игнорируется на прогулке `relative_to` (якорь фиксирует одну конкретную сессию), как и любой другой фасет.

**Покрытие детекции сабагентов (внутренняя нормализация парсеров, один публичный критерий):** **Claude** — layout каталога `subagents/` + sidechain-поле `parentUuid`; **OpenCode** — `session.parent_id`; **Codex** — `session_meta.payload.thread_source == "subagent"` + `parent_thread_id` (включая fallback на вложенный `source.subagent.thread_spawn.parent_thread_id`); **Pi** — заголовочное поле `parentSession`. **Antigravity** — в формате нет сигнала о родителе → всегда `kind="agent"`, шумом не бывает. Warmup/заготовки шумом **не** считаются: ни один формат агента не несёт для них надёжного дешёвого маркера, а эвристика по заголовку промахивалась бы — критерий остаётся точным (шум == сабагент), а не угаданным.

## Источники Claude-сессий (CLI + Desktop-оверлей)

Claude-парсер сканирует **два корня** и сливает их в один список сессий (F1.3):

- **CLI-корень** — `~/.claude/projects/<slug>/<uuid>.jsonl`: транскрипты (`$AI_R_HOME/.claude/projects`, когда задан `AI_R_HOME`).
- **Desktop-корень** — `~/.config/Claude/claude-code-sessions/<device>/<workspace>/local_*.json` (`$AI_R_HOME/.config/Claude/claude-code-sessions` под `AI_R_HOME`): собственное хранилище приложения Claude **Desktop**. В нём **только метаданные** — по одному JSON-объекту на сессию (`sessionId`, `cliSessionId`, `title` + `titleSource`, `cwd`, таймстампы в epoch-мс, `model`, `permissionMode`), НЕ транскрипты: транскрипт Desktop-сессии по-прежнему лежит в CLI-корне под `cliSessionId`.

**Правила слияния:** ключ дедупа — uuid сессии (`cliSessionId` == имя CLI-JSONL) — сессия, видимая в обоих корнях, возвращается ОДИН раз, обогащённой: побеждает Desktop-`title` (это заголовок, который пользователь видит в приложении, а значит и ищет; CLI-заголовок сохраняется в `extra["cli_title"]`). Происхождение помечается в `extra["source_root"]`: `"desktop"` = сессию вели из Desktop-приложения (сигнал *поверхности запуска (launch surface)*; F1.4 отдаёт его first-class как `launch_surface="claude-desktop"`), `"cli"` = обычная CLI-сессия (`launch_surface="claude-cli"`). Uuid, известный ТОЛЬКО Desktop-хранилищу (транскрипт удалён), всё равно появляется как **reference-only** сессия — `message_count == 0`, чтение сообщений возвращает пустой список (честный ответ, не ошибка), `path` указывает на JSON метаданных. Отсутствующий корень пропускается, никогда не ошибка. Оверлей действует единообразно на `list_sessions` / `read_session` / `search` / `session_exists`; `source_roots()` отдаёт оба корня, чтобы диагностика пустых результатов могла их назвать. Заметка для герметичных тестов: явный `base_dir` **без** явного desktop-корня ограничивает скан CLI-корнем — фикстурные вызовы никогда не подтягивают реальный HOME.

## Происхождение сессии (`project_dir` + `launch_surface`)

Каждая сводка сессии (`list_sessions` / `read_session` / кандидаты `search_sessions`) несёт два first-class поля происхождения рядом с `kind`/`parent_uuid`; оба `null`, когда в исходном формате нет сигнала — **отсутствие честно, ничего не выдумывается**:

- **`project_dir`** — директория проекта, в которой шла сессия. Сигнал по агентам (внутренняя нормализация парсеров, одно публичное поле): **Claude** — `cwd` уровня записи в CLI-транскрипте; фолбэк — `cwd`/`originCwd` из Desktop-метаданных (оверлей F1.3), затем **проверенное по файловой системе** декодирование slug-кодировки хранилища `projects/<slug>` (slug сплющивает `/` и `.` в `-`, поэтому дефис внутри реального имени неоднозначен — декодер перебирает возможные границы сегментов и принимает только путь, реально существующий как директория; непроверяемо → `null`, без угадывания). **Codex** — `session_meta.payload.cwd`. **OpenCode** — колонка `session.directory` (старые БД без колонки деградируют в `null` через legacy-SELECT-фолбэк, перечисление не ломается). **Pi** — `cwd` из заголовка сессии. **Antigravity** — в формате нет структурного поля cwd/директории → всегда `null`.
- **`launch_surface`** — конкретная поверхность, с которой вели сессию, только там, где данные позволяют её различить: **Claude** — `"claude-cli"` | `"claude-desktop"` (из сигнала Desktop-оверлея F1.3). **Codex** — сырая строка `session_meta.payload.originator`, передаётся дословно (наблюдалось: `"codex_vscode"`, `"Codex Desktop"`; никакой выдуманной таксономии поверх сырого значения). **Antigravity** — `"antigravity-ide"` | `"antigravity-cli"` (в каком brain-корне лежит сессия: `~/.gemini/antigravity/brain` — IDE-приложение, `~/.gemini/antigravity-cli/brain` — CLI). **OpenCode**/**Pi** — сигнала в формате нет → всегда `null` (колонка `agent` у OpenCode — это *режим* (plan/build), не поверхность).

**Фильтр `project_dir`** в `list_sessions` и `query`: оставляет только сессии, чей `project_dir` равен заданному пути **или является его потомком** — с учётом границы пути (`/a/b` матчит `/a/b` и `/a/b/sub`, но никогда соседний `/a/bc`), хвостовые слэши игнорируются, другой нормализации нет (`~`/`..`/симлинки сравниваются как записаны). Выбрано вместо точного совпадения, потому что «сессии этого проекта» должны включать сессии, начатые в поддиректории корня проекта. Сессии с `project_dir=null` никогда не матчатся (отсутствие сигнала — не wildcard). Применяется на уровне *сессии* до чтения сообщений (как `noise`), сочетается с остальными фильтрами по AND, игнорируется на проходе `relative_to`, а пустое/пробельное значение фильтра падает громко (`invalid_argument`). SSOT семантики: `src/ai_r/parsers/_common.py::project_dir_matches`.

## Команда продолжения (`resume_command`)

Каждая сводка сессии несёт `resume_command` (F2.2, рядом с `project_dir`/`launch_surface`): точную shell-команду в одну строку, которая заново открывает разговор в CLI его агента, или `null`, когда реальной команды не существует — **отсутствие честно, никогда не выдумывается; поле — только текст, ai-r никогда его не выполняет**. Команды сверены с собственным `--help` установленных CLI, не изобретены:

| агент | команда | почему такая форма |
|---|---|---|
| Claude | `cd <project_dir> && claude --resume <uuid>` | `claude --resume` ищет id в хранилище проекта *текущей рабочей директории* → префикс `cd` делает команду рабочей из любого шелла; при `project_dir=null` отдаётся голая `claude --resume <uuid>` (сработает только изнутри исходной директории проекта). **Reference-only Desktop-сессия** (транскрипт удалён, F1.3) → `null` — продолжать нечего. |
| Codex | `cd <project_dir> && codex resume <uuid>` | `codex resume <SESSION_ID>` адресуем по id в глобальном хранилище (cwd-фильтр влияет только на интерактивный picker); префикс `cd` держит продолженную сессию в её исходной директории (Codex всегда пишет `cwd` → префикс есть всегда). |
| OpenCode | `cd <project_dir> && opencode --session <id>` | флаг основной команды `-s, --session` («session id to continue»); TUI привязан к проекту → префикс `cd`, когда `session.directory` известна (старые БД без колонки → голая команда). |
| Pi | `cd <project_dir> && pi --session <path>` | `pi --session <path\|id>`: поиск по *id* ограничен session-директорией текущего проекта, а записанный **путь к файлу** сессии однозначен откуда угодно → отдаётся форма с путём. |
| Antigravity | `null` | сессии — brain-директории IDE без CLI-верба продолжения; `--resume` локального gemini CLI адресует **его собственное** хранилище по индексу/`latest`, а не id brain-директорий → реальной команды не существует. |

Кросс-агентные правила: **subagent-сессии (sidechain) никогда не продолжаемы** (`kind="subagent"`/установленный `parent_uuid` → `null` — CLI продолжают top-level интерактивные разговоры, а не порождённые tool-треды); каждое подставляемое значение (uuid/путь/директория) экранируется для шелла. SSOT семантики: `src/ai_r/resume.py::resume_command`.

## Границы вывода и исход tool-вызова

**Ограниченный вывод (untrusted-сессии бывают огромными — поверхность никогда не отдаёт неограниченные байты):** `find_tool_calls` режет у каждой записи поля `input`/`assistant`/`intent`/`output` (превышение обрезается маркером `…[truncated]` и перечисляется в per-record `truncated_fields`) и прекращает добавлять записи при достижении общего байт-бюджета ответа, выставляя `output_truncated`; это отдельно от count-based `truncated` (есть ещё записи). `get_body` ограничивает тело через `max_chars` (`body_truncated`). Tool-input больше 1 МБ никогда не JSON-декодится (возвращается как есть) — общий guard и на событийном потоке, и в `find_tool_calls`. `read_session` рендерит результат вызова как `[tool_result ok: <snippet>]` или `[tool_result ERROR: <snippet>]` (был голый `[tool_result]`).

**Адаптивная обрезка вывода (`output_mode`):** cap на поле `output` каждой записи — `_OUTPUT_CHARS_CAP = 2000` символов. Как тратится этот бюджет, задаёт `output_mode ∈ {"head", "tail", "smart"}`. Дефолт (`output_mode=None`) — **адаптивно per record**: запись с `is_error == True` обрезается по `"smart"` (surface строк-ошибок — `error`/`fatal`/`traceback`/… — плюс хвост, чтобы ошибка в *конце* длинного лога не терялась при head-обрезке), а успешная запись — по `"head"` (legacy-поведение). Явный `output_mode` форсит одну стратегию для всех записей. `smart`/`tail` могут вернуть до ~2× cap, чтобы уместить и поднятые строки, и хвост; при любой обрезке `output` по-прежнему перечисляется в `truncated_fields` этой записи.

**Фильтрация `find_tool_calls` (все опциональны, композируются по И):** помимо `tool_name`/`tool_name_pattern` записи сужаются через `input_contains` (case-insensitive substring по сериализованному tool-input / тексту команды), `output_contains` (ci substring по коррелированному `output`), `output_excludes` (отбросить запись, чей `output` содержит маркер — заданный вызывающим фильтр шума, напр. строка harness security-гейта, `"user rejected"`, `"MANUAL COMMIT BLOCKED"`; **такого списка в ядре НЕ зашито**) и `is_error` (tri-state: `None` = все, `True` = только ошибки, `False` = только успех). Все фильтры пересекаются (И). Отдельного verb'а «ошибка + домен» **нет**: эта связка — *композиция*, напр. `find_tool_calls(input_contains="git", is_error=True)` возвращает реальные сбои команд выбранного домена (`git` — лишь пример домена, не спец-случай).

**`is_error` (исход tool-вызова) — cross-agent best-effort:** **Claude** и **OpenCode** несут реальный флаг успех/ошибка (у Claude — `tool_result.is_error`; у OpenCode — `state.status == "error"`). **Codex** и **Pi** не имеют поля ошибки в записях результата → `is_error` всегда `False` (отсутствие флага, не доказательство успеха). **Antigravity** вообще не эмитит tool-result-записей → сигнала исхода нет. Консьюмеры НЕ должны читать cross-agent `is_error=False` как «подтверждённый успех» для Codex/Pi/Antigravity. `find_tool_calls` теперь несёт тот же `is_error` в каждой записи плюс коррелированный `output` (содержимое tool-result, с char-cap) — корреляция по tool_use_id (у Claude `tool_use.id` / у OpenCode `callID`); с той же best-effort-оговоркой (`is_error` авторитетен только для Claude/OpenCode и по умолчанию `False` для Codex/Pi/Antigravity либо когда результат не коррелируется). Чтобы эта честность была машиночитаемой, каждая запись `find_tool_calls` дополнительно несёт `is_error_reliable` (bool): `True` для Claude/OpenCode (значение подкреплено реальным флагом), `False` для Codex/Pi/Antigravity (источника нет → `is_error` всегда `False` и может **недосчитывать** настоящие сбои). Консьюмер, фильтрующий `is_error=True`, должен читать `is_error_reliable`, чтобы понять, означает ли `False` «подтверждённый успех» или лишь «нет сигнала».

## Редакция секретов (маскирование на выводе)

Реальные транскрипты сплошь и рядом содержат вставленные секреты, поэтому **каждый метод, отдающий текст из сессий, по умолчанию маскирует их на выводе** (F2.1). Отдающие поверхности: `query` (`text`/`intent`), `get_body` (`text`/`body`/`title`/`steps`), `plan` (title/steps/refs), `diff` и `session_diff` (текст диффа/hunks/intents), `read_session` (заголовок + содержимое сообщений), `search_sessions` (title/snippet), `list_sessions` (title), `find_file_edits` и `find_tool_calls` (title/intent/assistant/input/output). Каждая замена — `[REDACTED_<TYPE>]`; если что-то замаскировано, ответ несёт дикт-счётчик `redactions` тип→число (отсутствует = ничего не маскировалось). `redact=false` на любом из этих методов возвращает сырое содержимое. `session_stats`/`aggregate` отдают только счётчики/метки, производные от rows, которые вызывающий уже держит — собственного текста сессий нет, поэтому параметра `redact` нет; `detect_current` читает runtime-окружение, не транскрипты.

**Типы:** `PRIVATE_KEY` (PEM-блоки), `AWS_KEY`/`AWS_SECRET`, `GITHUB_TOKEN`, `GITLAB_TOKEN`, `ANTHROPIC_KEY`, `OPENAI_KEY`, `SLACK_TOKEN`, `URL_CREDENTIALS` (`user:pass@` в URL — маскируется только сам credential-спан), `BEARER_TOKEN` (только значение, префикс `Bearer` сохраняется), `GENERIC_SECRET` (явное секрет-подобное имя ключа — `password`/`token`/`api_key`/… — с токен-образным значением). SSOT паттернов: `src/ai_r/redact.py`. **Защита от ложных срабатываний:** uuid'ы, git-хеши и проза вроде `sk-learn` или `Bearer authentication` никогда не срабатывают (value-паттерны требуют цифру; generic catch-all требует имя ключа + `:`/`=`); честный компромисс — полностью буквенный пароль под generic-ключом не маскируется.

**Только на выводе (emission-time only):** редакция никогда не касается сканирования и матчинга — любой фильтр (`text`, `input_contains`, поисковые запросы, …) матчится по СЫРОМУ хранимому тексту, поэтому поиск буквального секрета всё равно находит его сессию (маскируется только показываемый вывод), а плейсхолдер `[REDACTED_*]` никогда не может сматчиться как поисковый термин. Диагностика пустых результатов говорит об этом явно: значение фильтра, которое ЯВЛЯЕТСЯ плейсхолдером или само похоже на секрет, получает подсказку, что редакция действует только на вывод, с предложением `redact=false` для сырого вывода.

## Пустые результаты и поиск сессии

**Диагностика пустого результата (ответ с нулём результатов объясняет себя сам, а не голый пустой список):** когда сканирующий метод — `query`, `search_sessions`, `find_tool_calls`, `find_file_edits`, `list_sessions` — ничего не находит, ответ несёт объект `diagnostics` рядом с пустым списком/счётчиком. Форма: `scanned` (по записи на каждый просканированный агент — счётчик `sessions`, `date_min`/`date_max`, `source_found`, плюс per-agent `hint`, например `source not found: ~/.pi/agent/sessions` или `source present but contains no sessions`), `corpus` (всего сессий + общие границы дат), `filters` (эхо активных фильтров), `hints` (кандидаты-причины: граница `since`/`until`, отсёкшая весь корпус, называется явно — напр. `since='2030-01-01' is after the newest session (…) — the date filter excludes the entire corpus`; иначе перечисляются оставшиеся фильтры либо результат объявляется настоящим no-match). Диагностика считается только на пустом пути — непустой ответ её не несёт (и не платит за неё) — и никогда не роняет ответ (нечитаемый источник деградирует в per-agent hint).

**`read_session` больше не требует `agent`:** параметр опционален. Без него id ищется по всем парсерам (id сессий на практике уникальны между агентами). Редкая коллизия id между агентами возвращает `{ambiguous: true, candidates: [...], count}` — список для развязки (disambiguation), где каждый кандидат несёт свой `agent`, НЕ ошибку; повторите вызов с явным `agent`. Промах возвращает `{error: "not_found", agents_scanned: [...]}`. `get_body` и так был agent-free (его event id содержит сессию-владельца).

**Контракт ошибок CLI (скрипт-потребитель никогда не видит трейсбек питона):** ожидаемые сбои сохраняют одну строку `ai-r: <message>` на stderr + ненулевой exit (1 общий / 2 неоднозначность или неверный аргумент / 3 не найдено); *неожиданная* внутренняя ошибка выдаётся одной структурированной JSON-строкой на stderr (`{"error": "internal_error", "type", "message", "hint"}`) с exit-кодом 1. `AI_R_DEBUG=1` перебрасывает исходное исключение для отладки.

### Событийное ядро

Вербы выше — новые: одно **событийное ядро** заменяет кучу разовых
инструментов. Каждый парсер читает логи одного агента и выдаёт типизированные
модели, которые нормализуются в единый агент-нейтральный поток —
`user_turn` / `assistant_turn` / `tool_call(...)` / `plan_event`. Небольшой
набор вербов фильтрует, агрегирует и diff-ит этот поток; различия агентов
(`ExitPlanMode` против `update_plan` против `implementation_plan.md`) скрыты
внутри парсеров — вызывающий видит одну форму.

Честно про границу: это **только извлечение сущностей** — реплики, вызовы
инструментов, планы, намерения, реакции. Это **не** граф и **не** хранилище
памяти. Что делать дальше (граф знаний, Obsidian, постоянная память) — уже на
твоей стороне, вне этого репозитория. Полную слоёную схему и список
MCP-инструментов см. в [docs/architecture.md](./docs/architecture.md).

## Быстрый старт (1 команда)

Требования: Python 3.11+ с `venv` или `pip`, и `jq` (для автопатча MCP-конфигов
Claude и Antigravity — остальным `jq` не нужен).

```bash
git clone https://github.com/pro-target/ai-r.git ~/dev/ai-r
cd ~/dev/ai-r && bash install.sh
```

Установщик создаёт venv, ставит runtime-пакет, патчит MCP-конфиги для
**Claude**, **Codex**, **OpenCode**, **Antigravity** (где конфиги существуют),
ставит CLI-скилл **Pi** и прогоняет смоук-тесты.

Необязательное дополнение — `tokens`: `AI_R_EXTRAS=tokens bash install.sh`
(или `pip install "ai-r[tokens]"`) добавляет
[tiktoken](https://github.com/openai/tiktoken) — более точные **оценки**
токенов для сессий, чей формат не хранит точных чисел расхода. Полностью
опционально: без него точные числа по-прежнему берутся прямо из файлов сессий
(где записаны), а оценка деградирует до грубой эвристики «символы/4» с честной
пометкой `estimate` — никогда не падение.

Необязательное дополнение — `semantic`: `AI_R_EXTRAS=semantic bash install.sh`
(или `pip install "ai-r[semantic]"` + разовое скачивание модели, которое
установщик делает сам) включает `sort="semantic"` в текстовом поиске (`query`,
`search_sessions`): топ-50 кандидатов от BM25 пересортируются по **смыслу**
локальной многоязычной моделью эмбеддингов (embeddings) —
[intfloat/multilingual-e5-small](https://huggingface.co/intfloat/multilingual-e5-small)
(int8 ONNX, ~118 МБ, MIT), напрямую через
[onnxruntime](https://onnxruntime.ai) + [tokenizers](https://github.com/huggingface/tokenizers),
без torch и без постоянного индекса. Почему эта модель: сильный межъязыковой
поиск (русский запрос находит английскую сессию и наоборот) при малом размере.
Как устроен балл, простыми словами: BM25 отбирает 50 лучших совпадений по
словам (это бюджет затрат, а не отсечка по качеству — порога схожести
намеренно НЕТ: у этого семейства моделей даже несвязанные тексты получают
≈0.7); внутри пула итоговый балл = **75 % смысл + 25 % совпадение слов** —
смысл главнее, а словесная доля не даёт утопить точное вхождение термина и
разбивает ничьи. Полностью опционально: без пакетов или файлов модели
`sort="semantic"` честно откатывается к порядку BM25, и ответ объясняет
почему (`semantic: {active: false, reason, fallback: "bm25"}`) — никогда не
падение.

## Границы: читалка, не охранник

- **Только чтение.** Никогда не запускает код агента и не пишет в его историю —
  читает и возвращает.
- **Ни графа, ни памяти.** Достаёт сущности (реплики, вызовы, планы,
  намерения). Строить из них граф знаний или память — твоя задача, не его.
- **Не контроль доступа.** Кто дотянулся до CLI, MCP-сервера или пакета — читает
  любую сессию. Проверки прав перед парсерами нет; держи там, куда чужие
  локальные процессы не дотянутся.
- **Содержимое сессий — данные, не команды.** Кто читает (аудитор,
  суммаризатор), обязан относиться к тексту сессии как к данным, а не
  инструкциям. См. [Безопасность](docs/security.md).

| `resume_command` (summary field) | 1 | Every session summary carries `resume_command` — the ready-to-run CLI one-liner (`cd <project_dir> && claude --resume <uuid>` / `codex resume <uuid>` / `opencode --session <id>` / `pi --session <path>`), shell-quoted, `cd`-prefixed when `project_dir` is known; `null` exactly where no real command exists (Antigravity, subagent sessions, reference-only Desktop sessions) — text only, never executed. |

| `read_session` | 3 | Reads one session into the compact `{role, content}` projection with metadata + pagination echo; `offset`/`limit` page a stable ordered list, `total` invariant across slices; `agent` is **optional** — an id resolves across every parser, a rare cross-agent id collision returns a `candidates` list (not an error), a miss names `agents_scanned`. |
| `search_sessions` | 4 | Title/body/all scope; `AND` default, `OR` widens (`AND ⊆ OR`), negative `-term` excludes, quoted phrase is contiguous; `scope=body` returns a matching `snippet`; BM25 vs date sort; `noise=exclude` removes subagent matches before scanning, `noise=only` searches only the subagent tree. |
| empty-result diagnostics (cross-cutting) | 2 | A zero-result `query`/`search_sessions`/`find_tool_calls`/`find_file_edits`/`list_sessions` response carries `diagnostics` (per-agent scan counts + date bounds + `source_found`, corpus totals, cause hints: missing source dir / all-excluding `since`/`until` / remaining filters); a non-empty response never carries it. |
| secret redaction (cross-cutting) | 3 | Every text-emitting method masks secrets on output as `[REDACTED_<TYPE>]` by default and carries a `redactions` type→count dict; `redact=false` returns the raw content; matching always runs on the RAW stored text (searching a literal secret finds its session, only the display is masked); a `[REDACTED_*]` placeholder or secret-looking filter value on an empty result earns a diagnostics hint suggesting `redact=false`. |
| semantic sort (cross-cutting) | 3 | `sort="semantic"` on the text-search surface (`query` text facet, `search_sessions`) re-ranks the BM25 top-50 candidates by meaning with a local multilingual embedding model (`intfloat/multilingual-e5-small`, int8 ONNX, direct onnxruntime+tokenizers, mandatory `query:`/`passage:` prefixes applied internally, no persistent index); blended candidate score = 75 % meaning + 25 % word match (min–max normalized within the pool), no similarity cut-off — results are re-ordered, never dropped, the tail keeps BM25 order; the response carries a `semantic` report (`active: true` + model/candidates/weight, or `active: false` + plain-words `reason` + `fallback: "bm25"`); without the optional `ai-r[semantic]` deps/model files the order honestly falls back to plain BM25 — never a crash — and the default sorts never touch the module; cross-lingual ru↔en retrieval works both ways. |
| CLI error contract | 1 | A failing `ai-r` CLI invocation exits non-zero with a structured error on stderr (single `ai-r: …` line, or one JSON `internal_error` line for unexpected failures) — never a Python traceback; `AI_R_DEBUG=1` re-raises for debugging. |
